# TrendFetch PRD

## 1. 项目概述

### 1.1 项目名称
TrendFetch

### 1.2 项目目标
用户输入一个或多个搜索诉求，例如：

- `成人用品`
- `情趣用品店铺`
- `adult products wholesale`
- `sex toys online shop`

系统通过浏览器自动在 Google 中搜索这些关键词，自动翻页遍历搜索结果，筛选出与成人用品/情趣用品强相关的线上店铺站点，进入每个站点抓取并结构化保存以下信息：

1. 店铺名称
2. 网页链接
3. 联系人
4. 国家 - 城市
5. 联系电话
6. 电子邮箱

最终将结果导出为本地 CSV 文件。

补充要求：当用户输入一个或多个搜索诉求后，系统需要先将每个诉求自动扩展为多语言查询词，再分别执行搜索，以扩大覆盖面并提升跨区域店铺发现能力。

### 1.3 长期目标
在完成可运行项目后，将该流程沉淀为一个可复用的 Codex skill，用于后续快速执行同类“搜索引擎驱动的店铺线索采集”任务。

## 2. 背景与价值

### 2.1 背景
用户希望从 Google 搜索结果中系统化采集成人用品相关线上店铺的公开联系信息，而不是手动逐页浏览、人工记录。

### 2.2 业务价值

- 提高线索搜集效率
- 统一数据结构，便于后续筛选和外联
- 降低人工漏记、误记风险
- 为后续拓展到其他行业类目提供通用 SOP

## 3. 用户故事

### 3.1 核心用户故事
作为用户，我希望输入一组搜索关键词和采集参数，系统自动：

1. 打开浏览器并执行 Google 搜索
2. 自动翻页抓取搜索结果
3. 判断结果是否为成人用品/情趣用品相关线上店铺
4. 进入目标网站抓取联系信息
5. 去重并导出 CSV

### 3.2 扩展用户故事

- 作为用户，我希望限制搜索页数，避免跑太久
- 作为用户，我希望知道哪些结果被过滤掉，以及原因
- 作为用户，我希望失败站点被记录，便于复查
- 作为用户，我希望后续把这套流程复用成 skill，而不是每次重新写自动化

## 4. 产品范围

### 4.1 本期必须完成

- 支持 Google 搜索单个或多个关键词
- 支持自动将用户输入扩展为主要语言的等价搜索词
- 支持自动翻页
- 支持提取每页搜索结果标题、链接、摘要
- 支持基于规则筛选“成人用品/情趣用品相关线上店铺”
- 支持打开目标站点并抓取联系信息
- 支持结果去重
- 支持导出本地 CSV
- 支持基础运行日志
- 支持失败重试与失败记录

### 4.2 本期不做

- 登录后页面抓取
- 需要验证码识别的复杂绕过
- 大规模分布式采集
- CRM 集成
- 自动发送邮件或自动外联
- 非 Google 搜索引擎适配
- 对站点进行深度商品级别抓取

## 5. 输入与输出定义

### 5.1 输入

#### 用户输入参数

- `queries`: 搜索关键词列表
- `auto_translate_queries`: 是否自动扩展为多语言搜索，默认开启
- `search_languages`: 搜索语言列表，默认覆盖主要 20 种语言
- `max_pages_per_query`: 每个关键词最多翻页数
- `top_results_limit_per_query`: 每个查询最多浏览的自然搜索结果条数上限
- `max_results_per_site`: 每个站点最多抓取页面数
- `page_open_timeout_ms`: 打开候选网页的超时时间，默认 10000 毫秒
- `page_load_timeout_ms`: 页面关键内容等待超时时间，默认 10000 毫秒
- `max_site_failures_before_skip`: 单站点允许的失败次数上限
- `output_dir`: CSV 输出目录
- `country_hint`: 可选，用于偏向某国家搜索结果
- `language_hint`: 可选，用于偏向某语言搜索结果
- `headless`: 是否无头运行

#### 示例输入

```json
{
  "queries": ["成人用品 情趣用品 店铺", "adult toys online store"],
  "auto_translate_queries": true,
  "search_languages": ["en", "ja", "zh", "es", "pt", "fr", "de", "it", "ru", "ar", "ko", "tr", "nl", "pl", "hi", "vi", "th", "id", "ms", "uk"],
  "max_pages_per_query": 5,
  "top_results_limit_per_query": 100,
  "max_results_per_site": 10,
  "page_open_timeout_ms": 10000,
  "page_load_timeout_ms": 10000,
  "max_site_failures_before_skip": 2,
  "output_dir": "./output",
  "country_hint": "US",
  "language_hint": "en",
  "headless": false
}
```

### 5.2 输出

#### 主输出
一个 CSV 文件，字段为：

- `store_name`
- `website_url`
- `contact_person`
- `country`
- `city`
- `phone`
- `email`
- `query_original`
- `query_translated`
- `query_language`
- `source_query`
- `source_google_page`
- `source_contact_page`
- `confidence_score`
- `notes`

#### 辅助输出

- 运行日志文件
- 失败站点列表
- 过滤结果记录
- 查询扩展记录

#### 失败站点记录建议字段

- `website_url`
- `source_query`
- `query_language`
- `failure_stage`
- `failure_reason`
- `attempt_count`
- `last_error_at`

### 5.3 多语言查询扩展

#### 默认语言范围
默认将用户原始诉求扩展为以下主要语言的搜索词：

- 英语 `en`
- 日语 `ja`
- 简体中文 `zh`
- 西班牙语 `es`
- 葡萄牙语 `pt`
- 法语 `fr`
- 德语 `de`
- 意大利语 `it`
- 俄语 `ru`
- 阿拉伯语 `ar`
- 韩语 `ko`
- 土耳其语 `tr`
- 荷兰语 `nl`
- 波兰语 `pl`
- 印地语 `hi`
- 越南语 `vi`
- 泰语 `th`
- 印尼语 `id`
- 马来语 `ms`
- 乌克兰语 `uk`

#### 扩展规则

1. 用户输入保留为原始查询，不被覆盖
2. 每个原始查询自动生成 20 个语言版本
3. 翻译结果应尽量贴近“搜索习惯用语”，而不是逐字直译
4. 每个语言版本保留语言标识，便于后续结果追踪与去重
5. 若用户显式指定 `search_languages`，则只按指定语言扩展
6. 多语言扩展后，每个语言查询都要受 `top_results_limit_per_query` 限制，防止无限搜索

#### 示例
用户输入：

- `成人用品`

系统扩展示意：

- `adult products`
- `sex toys`
- `アダルトグッズ`
- `大人のおもちゃ`
- `productos para adultos`
- `juguetes sexuales`
- `produtos adultos`
- `sex shop`

说明：实现时不应机械只保留一个译文。对于高价值行业词，允许维护“每种语言 1 到 3 个常用搜索表达”的映射表，以提升召回率。

## 6. 关键定义

### 6.1 什么算“线上店铺”
满足以下条件之一，可视为候选线上店铺：

- 官网中存在商品目录、购物车、批发目录、询盘入口、下单入口
- 页面明确销售成人用品、情趣用品、sex toys、adult toys、intimate products 等
- 页面不是纯资讯站、论坛、百科页、新闻页或单纯目录站

### 6.2 什么算“成人用品/情趣用品相关”
满足以下任一条件：

- 站点标题、描述、正文出现高相关关键词
- 导航、产品页、分类页明确出现相关类目
- about/contact/product 页面表明其主营相关产品

### 6.3 什么不应纳入结果

- Google 广告落地页但并非实际店铺官网
- 黄页/目录聚合站
- 论坛、博客、新闻报道
- 平台型搜索结果页，无法确认具体店铺主体
- 明显非成人用品业务站点

## 7. 端到端流程

### 7.1 搜索阶段

1. 接收用户原始查询
2. 将每个原始查询扩展为多语言查询集合
3. 启动 Playwright 浏览器
4. 逐个执行多语言 Google 搜索
5. 抓取当前页自然搜索结果
6. 判断是否存在下一页
7. 同时检查当前语言查询累计浏览结果数，达到 `top_results_limit_per_query` 后立即停止该查询
8. 按配置翻页直到达到页数上限、结果条数上限或无下一页
9. 记录每条结果对应的原始查询、翻译查询、语言代码

### 7.2 搜索结果预筛选阶段

对每条 Google 搜索结果执行预判：

- URL 域名是否可访问
- 标题/摘要是否包含行业关键词
- 是否明显为目录站、论坛、新闻站、社媒页

预筛选通过后，进入站点抓取阶段。

### 7.3 站点抓取阶段

1. 打开候选站点首页
2. 若 `page_open_timeout_ms` 内无法打开，或导航报错，则记录失败并跳过该站点
3. 判断站点是否为成人用品相关线上店铺
4. 抓取首页可见联系信息
5. 尝试访问：
   - `contact`
   - `about`
   - `support`
   - `wholesale`
   - `privacy`
   - `terms`
6. 对每个辅助页面都应用 `page_load_timeout_ms`，单页失败不终止整站流程
7. 从这些页面抽取联系人、电话、邮箱、国家、城市
8. 若首页无法确认店铺名称，则从 title、logo alt、about 页面补充
9. 若累计失败次数达到 `max_site_failures_before_skip`，则停止该站点后续访问并进入下一个结果

### 7.4 数据清洗阶段

1. 标准化电话格式
2. 标准化邮箱格式
3. 拆分国家与城市
4. 按域名去重
5. 合并同站点在不同语言查询下的重复命中
6. 保留所有命中来源语言与原始查询映射
7. 合并同站点多页面信息
8. 计算置信度

### 7.5 导出阶段

1. 生成 CSV
2. 生成失败记录
3. 输出运行摘要

## 8. 筛选规则设计

### 8.1 一级筛选：搜索结果层
基于标题、摘要、URL 关键词判断：

正向关键词示例：

- `adult`
- `sex toys`
- `pleasure`
- `intimate`
- `vibrator`
- `dildo`
- `lingerie`
- `adult shop`
- `erotic`

负向关键词示例：

- `news`
- `wiki`
- `reddit`
- `forum`
- `directory`
- `review`
- `blog`
- `quora`

### 8.2 二级筛选：站点层
从站点内容判断：

- 是否存在产品或类目页
- 是否存在购物车、询盘、批发、MOQ、catalog 等交易信号
- 是否明确表述主营成人用品

### 8.3 结果置信度
建议使用 0 到 1 的置信度分数，来源包括：

- 标题匹配强度
- 页面正文匹配强度
- 是否存在 contact/about 页面
- 联系方式完整度
- 是否存在交易信号
- 是否被多个语言查询重复命中

### 8.4 多语言去重策略

- 主键优先使用规范化域名
- 同域名下不同 URL 优先合并为同一店铺记录
- 若同一站点被多种语言查询命中，只输出一条主记录
- 主记录中保留 `query_original`、`query_translated`、`query_language` 的聚合痕迹
- 若同一公司存在多个国家站点，按不同主域名或明确地区站独立记录

## 9. 联系信息抽取规则

### 9.1 店铺名称
优先级：

1. 页头品牌名
2. 页面 title
3. about/contact 中的公司名
4. 页脚版权主体

### 9.2 联系人
优先从以下模式提取：

- `Contact:`
- `Sales manager`
- `Account manager`
- `Founder`
- `CEO`
- `Mr.`
- `Ms.`

说明：很多站点不公开联系人，因此该字段允许为空。

### 9.3 国家 - 城市
来源优先级：

1. contact 页面地址块
2. 页脚地址
3. about 页面公司介绍
4. schema.org / structured data

### 9.4 联系电话
来源：

- `tel:` 链接
- 页面正文正则匹配
- WhatsApp 联系号码

### 9.5 电子邮箱
来源：

- `mailto:` 链接
- 页面正文正则匹配
- 混淆邮箱解码后文本

## 10. 异常与边界情况

### 10.1 Google 层面

- 验证码或反爬页面
- 地区化结果差异
- 搜索结果 DOM 结构变化
- 广告结果与自然结果混杂
- 多语言查询量放大后，更容易触发风控

### 10.2 站点层面

- 页面加载慢
- Cloudflare/反机器人拦截
- 联系页不存在
- 联系方式在图片中
- 只有表单，没有直接邮箱/电话
- 单页应用需等待渲染
- 导航超时
- 证书错误、DNS 错误、连接中断

### 10.4 合理跳过机制

以下情况应视为“可跳过”，记录失败后继续处理下一个目标，而不是中断整个任务：

- 候选站点首页在 `page_open_timeout_ms` 内未打开
- 页面返回网络错误、DNS 错误、证书错误
- 页面状态异常，无法完成基础 DOM 加载
- 站点触发机器人拦截且重试后仍失败
- contact/about 等辅助页面单页超时或报错

建议策略：

1. 首页失败：最多重试 1 次，仍失败则整站跳过
2. 辅助页面失败：直接跳过该页面，保留已抓到的数据
3. 同站点累计失败达到阈值：终止该站点，继续下一个结果
4. 单 query 或单站点失败不得中断整个任务

### 10.3 数据层面

- 同一站点多个域名
- 联系方式缺失
- 国家和城市无法可靠拆分
- 邮箱和电话属于客服而非销售联系人

## 11. 非功能需求

### 11.1 稳定性

- 单站失败不影响整体任务
- 支持超时控制
- 支持有限重试
- 支持达到上限后主动停止搜索
- 支持错误分类后的跳过策略

### 11.2 可维护性

- 搜索逻辑、筛选逻辑、抽取逻辑、导出逻辑模块化
- 关键规则可配置
- 日志清晰可追踪

### 11.3 可观测性

- 记录每个 query 的页数、结果数、通过数、失败数
- 记录每个原始 query 扩展出的语言版本与执行状态
- 记录每个站点失败原因
- 支持 debug 模式保留 HTML 或截图

### 11.4 性能

- 可串行先跑通
- 后续可升级为受控并发
- 对同域名避免重复打开
- 搜索结果处理应支持“达到条数上限即停止”，避免无边界消耗

## 12. 合规与风控要求

### 12.1 数据范围
仅采集公开网页中可直接访问的信息，不尝试突破登录、付费墙或技术访问限制。

### 12.2 站点尊重策略

- 控制请求频率
- 限制并发
- 不进行大规模压测式访问

### 12.3 搜索引擎风险
Google 对自动化访问较敏感，因此必须预留以下策略：

- 浏览器真实行为模拟
- 节流与随机停顿
- 可切换手动登录态或人工介入模式
- 将“Google 搜索阶段”和“站点抓取阶段”解耦，必要时支持导入已采集 SERP 结果
- 支持分批执行语言查询，避免一次性触发 20 倍搜索量
- 支持语言轮转和任务断点续跑

说明：这部分不是可选项，而是项目能否长期可用的关键。

## 13. 推荐技术方案

### 13.1 首选方案
Node.js + Playwright

原因：

- 浏览器控制成熟
- 对 Google 搜索和站点动态页面兼容性较好
- 便于后续封装为脚本与 skill

### 13.2 备选方案
agent-browser 或基于浏览器代理的自动化框架

适用场景：

- 需要更强的自主导航与页面理解
- 需要在复杂页面中自主寻找 contact/about 入口

### 13.3 不建议首期依赖

- 纯 HTTP 抓取：Google 与动态站点稳定性差
- 重型多代理架构：首期复杂度过高

## 14. 建议模块拆分

### 14.1 核心模块

- `query_runner`: 执行 Google 搜索与翻页
- `serp_parser`: 解析搜索结果
- `candidate_filter`: 预筛选候选站点
- `site_crawler`: 打开候选站点并导航
- `contact_extractor`: 抽取联系信息
- `normalizer`: 标准化字段
- `deduper`: 去重合并
- `csv_writer`: 输出 CSV
- `run_logger`: 日志与失败记录

### 14.2 配置模块

- 查询参数配置
- 多语言词表与翻译映射配置
- 关键词词典
- 黑白名单域名
- 超时与重试参数
- 搜索结果条数上限配置
- 可跳过错误类型配置

## 15. 验收标准

### 15.1 功能验收

1. 输入至少 1 个 query 后可启动任务
2. 能自动扩展为多语言查询并执行搜索
3. 能按 `top_results_limit_per_query` 限制每个查询的最大浏览结果数
4. 能自动抓取至少前 N 页 Google 自然搜索结果
5. 能过滤明显无关结果
6. 能在页面 10 秒无法打开或出现导航错误时跳过站点并继续任务
7. 能从有效站点中抽取至少部分联系信息
8. 能生成本地 CSV

### 15.2 数据验收
以一个 50 个候选站点样本为基准：

- 与行业相关站点识别准确率目标 >= 80%
- 邮箱或电话至少命中其一的站点占比目标 >= 50%
- 重复记录率 <= 10%
- 多语言扩展后，独立有效站点召回量应明显高于单语言基线

### 15.3 工程验收

- 代码模块清晰
- 支持配置化运行
- 有基础文档和运行说明
- 有最小可用测试或回归验证脚本

## 16. 里程碑建议

### M1: MVP 跑通

- 单 query
- 多语言查询扩展
- 前 2 到 3 页 Google 结果
- 每个 query 的结果浏览上限
- 人工规则筛选
- 抓邮箱/电话/店铺名
- 基础超时跳过机制

### M4: Skill 化

- 把 SOP 抽象为 skill

- 提供可复用脚本与参考规则库

## 17. Skill 沉淀方案

### 17.1 Skill 目标
将 TrendFetch 的操作流程沉淀为一个面向 Codex 的 skill，使后续用户只需给出：

- 搜索关键词
- 页数上限
- 输出路径

即可触发标准化流程。

### 17.2 Skill 建议内容

- `SKILL.md`: 核心工作流、触发条件、执行边界
- `references/filters.md`: 行业关键词与黑白名单规则
- `references/extraction.md`: 联系信息抽取策略
- `scripts/run_trendfetch.*`: 执行入口脚本

### 17.3 Skill 触发描述建议
适用于以下请求：

- 从 Google 搜索结果收集某行业线上店铺联系信息
- 自动浏览搜索结果并提取商家联系人/邮箱/电话
- 将搜索引擎线索采集输出为 CSV

## 18. 当前建议决策

本项目建议按以下顺序推进：

1. 先确认 PRD 和字段定义
2. 再确定 MVP 技术栈为 Playwright
3. 先做规则驱动版本，不先引入复杂 agent
4. 跑通后再抽象成 skill

原因很直接：这个任务的核心难点不在“会不会打开网页”，而在“筛选标准是否稳定”和“联系信息抽取是否可复用”。如果这两部分先不定清楚，后面无论用 Playwright 还是 agent-browser，都会反复返工。

## 19. 待确认问题

以下问题会直接影响实现方案，建议在进入开发前确认：

1. Google 是否必须为唯一搜索入口，还是允许后续扩展 Bing 等备选？
2. 目标市场是否有限定国家或语言？
3. 是否接受“没有联系人姓名，但有邮箱和电话”的站点进入最终结果？
4. CSV 是否需要增加 `whatsapp`、`company_address`、`source_snippet` 等字段？
5. 每次任务预期规模是多少，例如 50、200 还是 1000 个站点？
6. 是否需要保留截图、HTML 快照，用于人工复核？
7. 多语言扩展时，每种语言是保留 1 个查询表达，还是允许 2 到 3 个行业常用表达以提升召回？
8. 默认每个语言查询的 `top_results_limit_per_query` 设为多少更符合你的预期，例如 50、100 或 200？

## 20. 结论

TrendFetch 的首期目标应定义为：

“基于 Google 搜索结果，自动发现成人用品相关线上店铺，并从公开网页中提取结构化联系信息，最终输出为可复用的本地 CSV。”

这个定义足够聚焦，能支持快速做出 MVP，也为后续 skill 化保留了明确边界。
